## Module Organisation

### Canonical module layout

Every reusable module should follow this structure:

```
modules/<module-name>/
в”њв”Ђв”Ђ main.tf          # Resources
в”њв”Ђв”Ђ variables.tf     # Input variables
в”њв”Ђв”Ђ outputs.tf       # Output values
в”њв”Ђв”Ђ versions.tf      # Required providers and terraform version
в”њв”Ђв”Ђ README.md        # Auto-generated by terraform-docs
в””в”Ђв”Ђ examples/        # Usage examples (optional but recommended)
    в””в”Ђв”Ђ basic/
        в””в”Ђв”Ђ main.tf
```

### Anti-patterns to flag

| Anti-pattern | What to look for | Severity |
|-------------|-----------------|----------|
| **God module** | A single module with 500+ lines in `main.tf` or 20+ resources of different types | high |
| **No modules at all** | All `.tf` files in the root with no `modules/` directory, 10+ resources | high |
| **Deeply nested modules** | Modules calling modules calling modules (3+ levels deep) | medium |
| **Inline module sources** | `source = "../../../some/path"` with long relative paths | medium |
| **Hardcoded values in modules** | Modules with hardcoded regions, account IDs, or environment names instead of variables | high |
| **No versions.tf** | Missing `required_providers` block or terraform version constraint | medium |

### Module granularity guidelines

| Scope | Module boundary |
|-------|----------------|
| Single resource with config | Too small вЂ” use the resource directly |
| Logical unit (e.g., VPC + subnets + route tables) | Correct вЂ” cohesive, reusable |
| Entire environment (VPC + EKS + RDS + IAM) | Too large вЂ” split into composable modules |

## Module Versioning

All modules вЂ” internal and external вЂ” must be version-controlled and pinned with semantic versioning.

### Internal modules

Internal modules should live in their own repository (or a dedicated `modules/` monorepo) and be consumed via versioned references:

```hcl
module "vpc" {
  source  = "git::https://github.com/org/terraform-modules.git//modules/vpc?ref=v1.2.0"
}
```

For a modules monorepo, tag each module independently using a `<module-name>/v<semver>` convention:

```
# Tags in the modules repo
vpc/v1.0.0
vpc/v1.1.0
eks-cluster/v2.0.0
rds/v1.3.1
```

Alternatively, if modules are published to a private Terraform registry, pin via the `version` argument:

```hcl
module "vpc" {
  source  = "app.terraform.io/org/vpc/aws"
  version = "~> 1.2.0"
}
```

### External (third-party) modules

Always pin external modules to an exact version or a pessimistic constraint:

```hcl
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "5.8.1"
}
```

Never use unversioned external sources (`source = "github.com/org/repo"` with no `ref` or `version`).

### Version constraint guidelines

| Constraint | Meaning | Use when |
|-----------|---------|----------|
| `"5.8.1"` | Exact pin | Maximum stability, manual upgrade only |
| `"~> 5.8"` | `>= 5.8.0, < 5.9.0` (patch flexibility) | Default вЂ” accept patch fixes automatically |
| `">= 5.0, < 6.0"` | Major-range pin | Trust the module to follow semver across minors |
| No constraint / no `ref` | Latest / HEAD | **Never use** вЂ” breaks reproducibility |

### Anti-patterns to flag

| Anti-pattern | What to look for | Severity |
|-------------|-----------------|----------|
| **Unpinned external module** | `source` pointing to a registry or git repo with no `version` or `ref` argument | critical |
| **Unpinned internal module** | `source = "../modules/vpc"` via relative path with no versioning strategy вЂ” changes propagate immediately and uncontrollably to all consumers | high |
| **Wildcard version constraints** | `version = ">= 0.0.0"` or `version = "*"` | critical |
| **Commit SHA pins without semver** | `ref=abc123def` with no corresponding tag вЂ” makes upgrades opaque | medium |
| **Stale pins** | Module pinned to a version 2+ major versions behind latest, with no documented reason | low |

### Internal module release workflow

Internal modules should follow a release process:

1. Changes are made via PR to the modules repo.
2. PR is reviewed and merged.
3. A semver tag is created on the merge commit (manually or via CI).
4. Consumers update the `ref` or `version` to the new tag.

This ensures changes are intentional, reviewed, and traceable.

## Stack Separation

A **stack** is an independently plannable/applyable unit with its own state file. A stack composes one or more versioned modules to build a concrete deployment.

### Stack strategies

| Strategy | Description | Best for |
|----------|-------------|----------|
| **Deployment stacks** | Each stack represents a complete deployment (customer, product, environment). It calls multiple modules to assemble the full infrastructure. | Multi-tenant platforms, per-customer infra, repeatable product deployments |
| **Layer stacks** | Each stack represents an infrastructure layer (networking, compute, data). Environments are subdirectories within each layer. | Single-product teams where layers have independent lifecycles |

Both strategies are valid. Choose based on what constitutes the natural unit of change:
- If you deploy **complete environments** for different customers/products в†’ deployment stacks.
- If you manage **shared layers** that evolve independently в†’ layer stacks.
- A hybrid is common: shared layers (networking, DNS) + deployment stacks per customer/product on top.

### Deployment stacks

Each stack composes modules to build repeatable infrastructure for a specific deployment:

Each stack's `main.tf` composes the same set of versioned modules with deployment-specific variables:

```hcl
module "vpc" {
  source  = "git::https://github.com/org/terraform-modules.git//modules/vpc?ref=v1.2.0"
  cidr    = var.vpc_cidr
  # ...
}

module "eks" {
  source     = "git::https://github.com/org/terraform-modules.git//modules/eks?ref=v2.1.0"
  vpc_id     = module.vpc.vpc_id
  subnet_ids = module.vpc.private_subnet_ids
  # ...
}

module "rds" {
  source     = "git::https://github.com/org/terraform-modules.git//modules/rds?ref=v1.5.0"
  vpc_id     = module.vpc.vpc_id
  subnet_ids = module.vpc.database_subnet_ids
  # ...
}
```

This keeps modules reusable and stacks composable вЂ” add a new customer by creating a new stack directory with its own tfvars.

### Anti-patterns to flag

| Anti-pattern | What to look for | Severity |
|-------------|-----------------|----------|
| **Monolith stack** | Single `terraform apply` manages all customers/environments/layers together | critical |
| **Flat stack directories** | Stacks dumped directly under `stacks/` with no account/region/environment hierarchy вЂ” unclear which account, region, and environment a stack targets | high |
| **Environment copy-paste** | Identical `.tf` files duplicated across stacks with only tfvars differing, but no shared modules | high |
| **No backend config** | Missing `backend.tf` or `backend` block вЂ” state defaults to local | critical |
| **Shared state file** | Multiple deployments or unrelated layers in a single state | high |
| **No remote state data sources** | Stacks reference each other via hardcoded values instead of `terraform_remote_state` or data sources | medium |
| **Snowflake stacks** | Deployment stacks that should be identical (same product, different customer) but have diverged with inline resource definitions instead of shared modules | high |

## State Isolation

Each stack must have its own isolated state. Verify:

1. Every stack directory has a `backend.tf` (or Terramate generates one) with a **unique** key/path.
2. State locking is enabled (DynamoDB for S3 backends, built-in for GCS/azurerm).
3. No two stacks share the same state bucket key.

### State key naming convention

The state key should mirror the stack's directory path:

```
<account>/<region>/<environment>/<stack-name>/terraform.tfstate
```

Examples:
- `acme-123456789012/eu-west-1/production/platform/terraform.tfstate`
- `shared-services-111111111111/eu-west-1/production/networking/terraform.tfstate`

## DRY Patterns

### Reducing duplication across environments

| Pattern | When to use |
|---------|-------------|
| **Shared modules + tfvars** | Default approach вЂ” modules define resources, tfvars differ per env |
| **Terramate code generation** | When managing 10+ stacks with shared patterns |
| **Terragrunt** | When you need DRY backend config and dependency orchestration without Terramate |
| **Workspaces** | Simple projects with identical resources across envs (use sparingly вЂ” poor isolation) |

### Anti-patterns to flag

| Anti-pattern | What to look for | Severity |
|-------------|-----------------|----------|
| **Copy-paste stacks** | Identical module calls in every env directory with only values changed, and no shared variable definitions | medium |
| **Workspaces for production isolation** | `terraform workspace` used to separate dev/staging/prod (single state backend, weak isolation) | high |

## Naming and Conventions

| Item | Convention |
|------|-----------|
| Module directories | `modules/<noun>` (e.g., `modules/vpc`, `modules/eks-cluster`) |
| Stack directories | `stacks/<layer>/<environment>` |
| Variable names | `snake_case`, descriptive (e.g., `vpc_cidr_block`, not `cidr`) |
| Output names | `snake_case`, prefixed with resource type when ambiguous |
| Resource names | `snake_case`, use `this` for the primary resource in a module |
| File names | `main.tf`, `variables.tf`, `outputs.tf`, `versions.tf`, `backend.tf`, `data.tf`, `locals.tf` |

## Checklist

Use this checklist when reviewing a Terraform repo. Each item maps to a potential finding.

- [ ] Reusable modules exist in `modules/` with standard layout (main/variables/outputs/versions)
- [ ] No god modules (single module managing unrelated resources)
- [ ] All external modules are pinned to an exact version or pessimistic constraint (`~>`)
- [ ] Internal modules are consumed via versioned git refs or registry versions, not unversioned relative paths
- [ ] No wildcard version constraints (`*`, `>= 0.0.0`)
- [ ] Internal modules follow a semver tagging and release workflow
- [ ] Stacks follow the `stacks/<account>/<region>/<environment>/<stack-name>/` directory hierarchy
- [ ] Stacks follow a clear strategy (deployment-based, layer-based, or hybrid)
- [ ] Each stack composes versioned modules вЂ” no inline resource definitions for things that should be shared
- [ ] Deployment stacks for the same product use the same modules (no snowflakes)
- [ ] State keys mirror the stack directory path
- [ ] Each stack has its own state file with a unique key
- [ ] State locking is enabled
- [ ] Environment differences are expressed via tfvars or code generation, not copy-paste
- [ ] No hardcoded account IDs, regions, or environment names in modules
- [ ] `versions.tf` exists with provider and terraform version constraints
- [ ] No deeply nested module calls (3+ levels)
- [ ] Cross-stack references use `terraform_remote_state` or data sources, not hardcoded values
